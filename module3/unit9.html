Artefacts

Cyclomatic Complexity’s Relevance Today

Cyclomatic Complexity is a method which allows developers to measure the amount of possible "paths" a piece of code (module, class, method, etc.) can take, with the purpose of determining complexity. It is self-explanatory that the more paths a function leads down, the more complex it is to read, change, or test. On the contrary however, Shepperd (1998) finds several issues with this method of thinking, namely that it is too abstract, difficult to prove, not observable equally in different contexts, and based on mostly intuition.
Despite the criticisms, I believe that Cyclomatic Complexity can serve to be useful in specific use-cases. An abnormally high Cyclomatic Complexity is likely to be a sign of the code being unnecessarily convoluted. It should, however, not be relied on as the sole metric, as complexity is abstract and impossible to prove mathematically.
With regards to OOP, Cyclomatic Complexity is relevant simply because OOP code still has a set number of paths. Complexity within OOP however should also observe how convoluted class name and usage is, whether polymorphism is present, reusability of code, etc. These aspects are not observed by Cyclomatic Complexity and therefore other metrics should also be observed.

References: Shepperd, M. (1988) A Critique of Cyclomatic Complexity as a Software Metric, Software Engineering Journal.
